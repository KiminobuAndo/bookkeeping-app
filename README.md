## 1. サービス概要

会計は現代ビジネスにおける「3種の神器」の一つであり、その基礎に位置するのが簿記である。  
（参考URL： https://renaissance-media.jp/articles/-/8142）

本サービスは、簿記の理解において重要な仕訳の中でも、  
「現金が借方・貸方のどちらに来るか」という一点に学習対象を絞っている。  
（参考URL：https://biz.moneyforward.com/accounting/basic/57800/）

簿記の仕訳を実践的に理解するための最小単位のWebアプリとして設計されている。

---

## 2. このサービスへの思い・作りたい理由

簿記の学習で挫折してしまう人を見ていると、  
仕訳を毎回考え込んでしまい、作業として機械的に処理できていないケースが多いように感じました。  
その結果、学習が進むにつれて負担が増し、途中で手が止まってしまうことが少なくないと考えました。

実際、簿記検定の学習を始めると、仕訳の学習段階で  
現金勘定、売掛金勘定、買掛金勘定、売上勘定、仕入勘定など、  
**使用する勘定科目が一気に増えます**。  
（参考：商工会議所簿記検定試験出題区分表 2021 年度に適用  
https://www.kentei.ne.jp/wp/wp-content/uploads/2021/03/2021_shokai1-3.pdf）

私自身も簿記の学習を進める中で、  
「どの勘定科目を使えばよいのか分からない」という状況に陥り、  
これが学習初期における大きなハードルになっていると感じました。

一方で、簿記の内容をできるだけ単純に捉えると、  
本質は「お金が増えたのか、減ったのか」を整理して記録することにあります。

そこで本サービスでは、仕訳を行う際の判断基準として  
**「現金が借方・貸方のどちらに来るか」**という一点に着目し、  
まずは現金の動きに集中して仕訳を処理できる状態を目指しています。

仕訳を理屈で理解する前に、  
手を動かしながら判断を固定することで、  
簿記学習における初期のつまずきを減らしたいと考え、本サービスを企画しました。

---

## 3. ユーザー層について

一般的な簿記学習サービスでは、仕訳問題において  
- 勘定科目の選択  
- 借方・貸方の判断  
- 金額の入力  

といった複数の判断を同時に求められることが多く、初学者にとっては学習負荷が高くなりがちです。

本サービスは、そのような従来型の学習方法とは異なり、  
**仕訳の判断軸を「現金が借方・貸方のどちらに来るか」という一点に限定**しています。

仕訳全体を理解させることを目的とするのではなく、  
まず「現金の増減」という最も基本的かつ重要な視点を固定し、  
仕訳を機械的に処理できる感覚を身につけることを重視しています。

本サービスは、既存の簿記学習アプリの代替ではなく、  
**それらを使う前段階に位置づけられる「前提となる学習アプリ」**として設計されています。

---

## 4. サービスの利用イメージ

ユーザーは、本サービス上で提示される取引の例に対して、  
「現金が増えたのか、減ったのか」を判断し、  
現金が借方・貸方のどちらに来るかを選択します。

問題はテンポよく繰り返し解ける構成となっており、  
仕訳を考え込まずに手を動かすことを重視しています。

このような問題演習を重ねることで、  
ユーザーは仕訳を理屈で考える前に、  
現金の動きを基準とした判断を自然に行えるようになります。

その結果、簿記の仕訳を一つひとつ悩まずに処理できる感覚が身につき、  
簿記学習の初期段階を無理なく進められるようになります。

具体的な操作の流れについては、画面遷移図を参照してください。

## 5. ユーザーの獲得について

本サービスは、これから簿記の学習を始める初学者に向けて、  
Web検索や学習系コンテンツ経由での自然流入を想定しています。

具体的には、  
「簿記 初心者」  
「仕訳 わからない」  
「借方 貸方 覚え方」  
といった、学習初期に生じやすい疑問を持つ層に対して、  
本サービスの存在を届けることを想定しています。

また、本サービスは簿記学習の最小単位に特化しているため、  
資格試験対策のメイン教材としてではなく、  
学習の導入段階や補助ツールとして利用されることを前提としています。

そのため、学習ブログやSNS等で  
「考えずに仕訳に慣れるためのツール」  
として紹介されることで、初学者が気軽に試せる導線を作りたいと考えています。

## 6. サービスの差別化ポイント・推しポイント

多くの簿記学習サービスは、  
勘定科目の説明や仕訳のパターン解説を中心に構成されており、  
初学者に対して理解を求める設計になっています。

そのため、学習の初期段階から  
「なぜそうなるのか」を考える必要があり、  
仕訳に慣れる前に学習の負担が大きくなりがちです。

本サービスの最大の差別化ポイントは、  
仕訳の中でも「現金が借方・貸方のどちらに来るか」という一点だけに  
学習対象を絞っている点です。

仕訳全体を理解させるのではなく、  
まず現金の動きを判断基準として固定し、  
反復的な問題演習によって  
仕訳を機械的に処理できる感覚を身につけることを重視しています。

このように、理解を前提とするのではなく、  
最初に手を動かして慣れる設計を採用している点が、本サービスの推しポイントです。

簿記学習の入口における心理的・学習的ハードルを下げ、  
次の学習段階へ進みやすくする点に、本サービスの独自性があります。

## 7. 機能候補

### MVPリリース時に作っていたいもの

#### 1. 認証・ユーザー
- ユーザー登録（メールアドレス／パスワード／表示名）
- ログイン／ログアウト
- ログイン必須（未ログインでは演習不可）

#### 2. メインページ（ダッシュボード）
- コース選択（MVPは「現金借方貸方コース」のみ）
- 「勉強スタート」ボタン
- 左下：他ユーザーの直近利用状況（直近プレイ履歴の一覧）
- 右下：ログインユーザーの直近履歴（自分の直近プレイ履歴）

#### 3. 演習（現金借方貸方コース）
- 開始前カウントダウン（3→2→1）
- 1セッション10問の出題
- 回答：借方／貸方の二択（MVPはクリック選択）
- 各問10秒タイマー + 残り時間表示（残り5秒以降で減点）
- 採点
  - 0〜5秒：5点
  - 6秒：4点
  - 7秒：3点
  - 8秒：2点
  - 9秒：1点
  - 10秒：0点
  - 誤答：0点
  - 10問合計 ×2 で100点換算
- タイムアップ時は自動0点で次の問題へ

#### 4. 結果・成績保存
- 結果画面：総合点（100点換算）／正答数／所要時間／各問ログ
- 成績の保存
  - セッション単位（Result）
  - 問題単位（ResultDetail：回答、正誤、秒数、点数）

#### 5. 管理機能（問題を自作する前提）
- 管理者ログイン（最低限）
- 問題CRUD（作成／編集／削除）
- コースCRUD（MVPは1コースでも拡張前提で用意）

---

### 本リリースまでに作っていたいもの（盛り込めるベース）

#### 1. UI/UX強化
- 回答UIをドラッグ&ドロップ化（「現金」を借方／貸方へ配置）
- 演習中の演出強化（残り3秒警告、連続正解の表示など）

#### 2. コース拡張（学習ステップの段階化）
- 売上（売掛金 → 現金回収）
- 仕入（買掛金 → 現金支払い）
- 「日商簿記3級合格コース」など複数コースの追加

#### 3. 3級コース向け機能（将来）
- 勘定科目の選択（選択肢）
- 金額の入力／選択
- 解説表示（MVPでは不要だが将来必須）

#### 4. 運用・実務寄り機能
- パスワードリセット
- 成績の可視化（推移、平均、弱点）
- 検索／ページネーション（コース・問題）
- 監視（エラー監視、ログ）
- 決済（Stripe：サブスク／教材パック）

## 8. 使用する技術スタック

### MVP（現実に作り切る前提）

#### フレームワーク・言語
- Ruby on Rails（Rails 7系を想定）
- Ruby（3.x系を想定）

#### データベース
- PostgreSQL（開発：PostgreSQL 15+ を想定）

#### インフラ／デプロイ
- Docker / Docker Compose（開発環境）
- Render（デプロイ先候補）

### 使用予定のライブラリ／技術（CRUD以外）

#### 認証
- devise（ログイン必須要件に対応）

#### フロント（タイマー／カウントダウン）
- Rails標準の Hotwire（Turbo + Stimulus）
- JavaScript（setInterval）による10秒タイマー
- hidden input 等で経過秒数をサーバへ送信し採点

#### 表示最適化
- partial（メインページの「全体履歴」「自分履歴」部品化）
- includes 等で N+1 問題を回避

#### テスト
- RSpec（モデル／システムスペック）
